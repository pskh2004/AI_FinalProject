 
# پایپ‌لاین استاندارد پیش‌بینی ریزش مشتریان (Telco Customer Churn)

این پروژه، یک پایپ‌لاین استاندارد و ماژولار مبتنی بر اصول **MLOps** برای پیش‌بینی ریزش مشتریان مخابراتی (Customer Churn) با استفاده از دیتاست **IBM Telco** است. هدف اصلی پروژه، طراحی یک فرآیند تکرارپذیر از مدیریت نسخه‌های داده، آموزش و ارزیابی چندین مدل یادگیری ماشین، ردیابی آزمایش‌ها با **MLflow** و در نهایت استقرار مدل برگزیده در قالب یک میکروسرویس کانتینرایز شده با **Docker** است.

---

## ۱. ساختار پوشه‌بندی پروژه

ساختار پروژه بر اساس اصول طراحی ماژولار نرم‌افزاری به شکل زیر پیاده‌سازی شده است:

```text
project/
├── data/
│   ├── raw_telco.csv       # دیتاست اولیه خام دریافت شده از IBM
│   ├── v1/                 # نسخه خام بدون دستکاری
│   ├── v2/                 # نسخه پیش‌پردازش و پاک‌سازی‌شده
│   └── v3/                 # نسخه بهینه‌سازی‌شده با مهندسی ویژگی
├── src/
│   ├── __init__.py
│   ├── data_loader.py      # بارگذاری داده‌ها و ذخیره‌سازی نسخه‌ها
│   ├── preprocessing.py   # پاک‌سازی و آماده‌سازی داده‌ها (v2)
│   ├── features.py         # مهندسی ویژگی‌ها و نرمال‌سازی (v3)
│   ├── train.py            # توابع آموزش و اعتبارسنجی متقاطع K-Fold
│   ├── evaluate.py         # ارزیابی تک‌مرحله‌ای روی تست و رسم Confusion Matrix
│   ├── mlflow_utils.py     # مدیریت ثبت آزمایش‌ها، متریک‌ها و مدل‌ها در MLflow
│   └── app.py              # میکروسرویس FastAPI جهت استنباط آنلاین مدل
├── run_pipeline.py         # اسکریپت هماهنگ‌کننده و نقطه شروع اجرای پایپ‌لاین
├── Dockerfile              # تعریف محیط مستقل جهت استقرار مدل
├── requirements.txt        # کتابخانه‌ها و وابستگی‌های پروژه
└── README.md               # راهنمای فنی پروژه (سند حاضر)
```

## ۲. استراتژی نسخه‌بندی داده‌ها (Data Versioning)

برای بررسی چگونگی تأثیر پیش‌پردازش بر رفتار الگوریتم‌ها، داده‌ها به سه نسخه تقسیم شده‌اند:
* **نسخه خام (v1):** استفاده مستقیم از داده‌های اولیه بدون تغییر ساختاری. (برای سازگاری مدل‌ها، کدگذاری متغیرهای دسته‌ای در فرآیند آموزشِ این بخش به شکل خودکار اعمال می‌شود).
* **پاک‌سازی داده (v2):** حذف ستون غیرضروری `customerID`، رفع مقادیر خالی یا فاصله در ستون `TotalCharges` و جایگزینی با میانه، و تبدیل متغیرهای دسته‌ای به عددی از طریق کدگذاری وان-هات (One-Hot Encoding).
* **مهندسی ویژگی (v3):** ساخت متغیرهای تعاملیِ مرتبط با کسب‌وکار (نظیر نسبت هزینه ماهانه به هزینه کل `Monthly_to_Total_Ratio` و متغیر سال‌های عضویت `Tenure_Years`) همراه با استانداردسازی متغیرهای عددی مستمر با `StandardScaler`.

## ۳. تحلیل تجربی و نتایج ارزیابی مدل‌ها (بر اساس گزارش MLflow)

با توجه به آزمایش‌های ثبت‌شده در سرور ردیابی MLflow (مطابق نمودارهای خروجی)، متریک‌های ارزیابی برای مدل‌های مختلف روی نسخه‌های داده به شرح زیر ثبت شده است:

### جدول مقایسه عملکرد مدل‌ها

| ردیف | نام مدل و نسخه داده | دقت (Accuracy) | معیار F1-score | صحت (Precision) | فراخوانی (Recall) | معیار ROC-AUC |
| :--- | :--- | :---: | :---: | :---: | :---: | :---: |
| ۱ | **CatBoost_v3** | **0.81** | **0.61** | **0.68** | **0.56** | **0.86** |
| ۲ | **XGBoost_v3** | **0.81** | 0.60 | 0.67 | 0.55 | 0.85 |
| ۳ | **Random_Forest_v3** | 0.80 | 0.57 | 0.65 | 0.51 | 0.85 |
| ۴ | **Logistic_Regression_v3** | 0.80 | 0.60 | 0.63 | 0.57 | 0.85 |
| ۵ | **CatBoost_v2** | 0.80 | 0.60 | 0.66 | 0.55 | **0.86** |
| ۶ | **XGBoost_v2 (اجرای مجدد)** | **0.81** | **0.61** | 0.65 | **0.58** | 0.85 |
| ۷ | **XGBoost_v2 (پایه)** | 0.80 | 0.60 | 0.65 | 0.56 | 0.85 |
| ۸ | **Random_Forest_v2** | 0.80 | 0.59 | 0.66 | 0.53 | 0.85 |
| ۹ | **Logistic_Regression_v2** | 0.80 | 0.60 | 0.65 | 0.56 | 0.85 |

### یافته‌های کلیدی حاصل از مقایسه آزمایش‌ها:
1. **بهترین عملکرد کلی:** مدل **`CatBoost_v3`** متعادل‌ترین عملکرد را در اکثر شاخص‌ها نشان می‌دهد. این مدل به بیشترین دقت (0.81)، بیشترین صحت (0.68)، بالاترین مقدار F1-score (معادل 0.61) و بیشترین سطح زیر منحنی راک (0.86) دست یافته است.
2. **تاثیر مثبت مهندسی ویژگی (v3):** بررسی نتایج الگوریتم CatBoost نشان می‌دهد که ارتقاء داده‌ها از نسخه `v2` به `v3` منجر به بهبود شاخص صحت (Precision) از 0.66 به 0.68 و شاخص F1-score از 0.60 به 0.61 شده است. این تغییر نشان‌دهنده کاهش هشدارهای اشتباه (False Positives) در تشخیص ریزش مشتری است.
3. **پایداری بالای شاخص ROC-AUC:** شاخص تفکیک‌پذیری کلاس‌ها (ROC-AUC) در تمامی مدل‌ها بسیار پایدار و قوی است؛ به طوری که مدل‌های خانواده بوستینگ (`CatBoost` و `XGBoost`) مقادیر پایدار 0.85 و 0.86 را ثبت کرده‌اند که توانایی بالای مدل‌ها در اولویت‌بندی احتمال ریزش مشتریان را تایید می‌کند.
4. **انتخاب نهایی جهت استقرار:** مدل **`CatBoost_v3`** به عنوان مدل منتخب نهایی جهت استقرار در بخش تولید و میکروسرویس استعلام انتخاب شد.

## ۴. نحوه نصب و راه‌اندازی پروژه

### نیازمندی‌ها
پیش از شروع، مطمئن شوید که پایتون نسخه‌ی ۳.۱۰ یا بالاتر و ابزار گیت روی سیستم شما نصب هستند.

### گام اول: کلون کردن مخزن و نصب نیازمندی‌ها
```bash
git clone <repository_url>
cd project
python -m venv venv
source venv/bin/activate  # در ویندوز: venv\Scripts\activate
pip install -r requirements.txt
```

### گام دوم: راه‌اندازی سرور MLflow جهت ردیابی فرآیندها
برای ردیابی تمام مراحل پایپ‌لاین، سرور ذخیره‌سازی محلی MLflow را در یک ترمینال مجزا اجرا کنید:
```bash
mlflow server --host 127.0.0.1 --port 5000
```
دسترسی به داشبورد مقایسه از طریق آدرس `http://127.0.0.1:5000` امکان‌پذیر خواهد بود.

### گام سوم: اجرای خودکار پایپ‌لاین سراسری
شما می‌توانید با استفاده از اسکریپت `run_pipeline.py` فرآیند آماده‌سازی، مهندسی ویژگی، ارزیابی متقاطع و ارزیابی تست را برای هر یک از نسخه‌های مورد نظر اجرا کنید:
```bash
# اجرای پایپ‌لاین روی نسخه v1 (خام)
python run_pipeline.py --version v1 --data_path data/raw_telco.csv

# اجرای پایپ‌لاین روی نسخه v2 (پاک‌سازی‌شده)
python run_pipeline.py --version v2 --data_path data/raw_telco.csv

# اجرای پایپ‌لاین روی نسخه v3 (مهندسی ویژگی)
python run_pipeline.py --version v3 --data_path data/raw_telco.csv
```

## ۵. استقرار مدل (Model Deployment)

جهت استقرار مدل برگزیده (`CatBoost_v3`) در محیط‌های عملیاتی، سرویس استنباط آنلاین مبتنی بر FastAPI و داکر طراحی شده است.

### روش اول: اجرای محلی وب‌سرویس (بدون داکر)
ابتدا متغیر محیطی آدرس مدل برگزیده خود را از روی Run ID موجود در MLflow تنظیم کنید و سپس سرور را فعال نمایید:
```bash
# در سیستم‌عامل‌های یونیکس/مک
export MODEL_URI="runs:/<YOUR_BEST_RUN_ID>/model"

# در ویندوز (CMD)
set MODEL_URI=runs:/<YOUR_BEST_RUN_ID>/model

# راه‌اندازی وب‌سرویس
uvicorn src.app:app --host 0.0.0.0 --port 8000
```

### روش دوم: استفاده از Docker (استاندارد کانتینرسازی)
بسته‌بندی پروژه و ارسال آن به محیط عملیاتی به صورت یک ایمیج ایزوله با دستورات زیر انجام می‌گیرد:

۱. **بیلد کردن ایمیج داکر:**
```bash
docker build -t telco-churn-service:v1 .
```

۲. **اجرای کانتینر سرویس‌دهی:**
محیط کانتینر را به پورت ۸۰۰۰ متصل کرده و شناسه اجرای مدل برگزیده را به آن معرفی کنید:
```bash
docker run -d -p 8000:8000 -e MODEL_URI="runs:/<YOUR_BEST_RUN_ID>/model" telco-churn-service:v1
```

۳. **تست و ارسال درخواست پیش‌بینی به سرویس:**
اکنون وب‌سرویس بر روی آدرس `http://localhost:8000/predict` آماده دریافت درخواست‌ها است. می‌توانید با ارسال داده‌هایی با ساختار زیر، پاسخ وضعیت احتمال ریزش مشتری را به صورت زنده دریافت کنید:

```bash
curl -X 'POST' \
  'http://localhost:8000/predict' \
  -H 'Content-Type: application/json' \
  -d '{
  "features": {
    "tenure": -1.28,
    "MonthlyCharges": -1.16,
    "TotalCharges": -0.99,
    "Monthly_to_Total_Ratio": 1.42,
    "Tenure_Years": -1.28,
    "SeniorCitizen": 0,
    "Partner_Yes": 1,
    "Dependents_Yes": 0
  }
}'
```

نمونه پاسخ سرور:
```json
{
  "prediction": 1,
  "churn_probability": 0.742
}
```
